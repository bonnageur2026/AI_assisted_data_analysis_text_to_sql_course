# AI 보조 데이터 분석과 Text-to-SQL 질문 설계 교육 과정

> Python 코딩 없이 파일 기반 EDA, SQL 독해, 질문 설계, AI 결과 검증을 배우는 실무형 교육 과정

---

## 제목에 관한 정직한 범위 설정

이 과정은 수강생이 SQL이나 Python을 직접 작성하게 하는 과정이 아니다. 또한 운영 데이터베이스에 실제로 쿼리를 실행하는 본격적인 Text-to-SQL 파이프라인 구축 과정도 아니다.

1차적으로 다루는 것은 AI 도구에 CSV/엑셀 파일을 올려 수행하는 **대화형 분석**, 즉 **파일 기반 EDA(Exploratory Data Analysis)**이다. Text-to-SQL은 **좋은 질문을 SQL이 풀 수 있는 형태로 설계하고, 생성된 쿼리를 읽고 검증하는 능력**에 초점을 둔다.

실제 DB 연동, 접근 권한, 로그 관리, API key 관리, 배포, 운영 환경 구축은 심화 과정의 범위다. 이 경계를 흐리지 않는 것이 과정 신뢰도의 핵심이다.

이 과정은 AI에게 분석을 맡기는 법만 배우는 과정이 아니다. 더 정확히 말하면, **AI가 수행한 분석을 사람이 질문하고, 검증하고, 해석하고, 실제 의사결정으로 연결하는 법을 배우는 과정**이다.

---

## 대상

이 과정은 다음과 같은 수강생을 대상으로 한다.

- Python이나 SQL을 본격적으로 배운 적은 없지만, 업무 데이터를 분석해야 하는 실무자
- ChatGPT, Claude 등 생성형 AI로 CSV/엑셀 데이터를 분석하고 싶은 사람
- AI가 만든 표, 그래프, SQL, 통계 결과를 검증하는 법을 배우고 싶은 사람
- 사내 데이터 분석 Agent의 가능성과 한계를 이해하고 싶은 기획자, PM, 비개발자
- 마케팅, 영업, 재무, 인사 등 부서 데이터를 다루는 현업 담당자
- 데이터 분석을 외주나 개발팀에만 맡기지 않고, 스스로 질문을 구체화하고 결과를 해석하고 싶은 사람

## 이 과정이 다루지 않는 것

이 과정은 입문형 실무 교육으로 설계되어 있으므로 다음 항목은 본격적으로 다루지 않는다.

- Python 프로그래밍 기초 문법
- SQL 실전 개발자 과정
- 운영 DB에 직접 접속하는 실DB Text-to-SQL 시스템 구축
- 데이터 엔지니어링 파이프라인 구축
- 통계학 이론의 엄밀한 수식 전개
- 머신러닝 모델 개발
- 사내 보안망 안에서의 LLM 시스템 배포
- BI 대시보드 운영 체계 구축

다만 위 주제들이 왜 필요한지, 본 과정의 실무 활용과 어떻게 연결되는지는 개념 수준에서 다룬다.

## 과정 개요

생성형 AI 데이터 분석 교육은 Python이나 SQL 문법 자체를 가르치기보다, **데이터 문해력, 좋은 질문 설계, AI 결과의 검증·해석 능력**에 초점을 둘 때 가장 효과적이다.

다만 한 가지 원칙을 분명히 한다.

> **“코딩을 몰라도 된다”는 “데이터 구조와 통계 개념을 몰라도 된다”는 뜻이 아니다.**

오히려 이 과정의 핵심 산출물인 **검증 능력**은 최소한의 독해 능력 없이는 성립하지 않는다. 읽지 못하는 쿼리를 검증할 수 없고, 기준을 모르는 통계를 판단할 수 없다.

따라서 본 과정은 두 가지를 명시적으로 구분한다.

| 구분 | 의미 |
|---|---|
| 위임 | SQL/코드 작성, 반복 계산, 표·그래프 초안 생성은 AI에게 상당 부분 맡긴다. |
| 통제 | 데이터 구조 이해, 질문 설계, 결과 검증, 해석, 보안 판단은 사람이 책임진다. |

사람이 통제하려면 **최소한의 독해력**이 필요하다. 여기서 독해력은 SQL을 능숙하게 작성하는 능력이 아니라, AI가 생성한 SQL을 읽고 “이 쿼리가 무엇을 세는지” 설명할 수 있는 능력이다. 통계도 마찬가지다. 복잡한 수식을 직접 유도하는 것이 아니라, 평균·중앙값·이상치·상관관계·표본 수·비교 기준을 묻고 왜곡을 알아차리는 능력이 중요하다.

- 업무 목적에 맞는 데이터 분석 Agent prototype을 no-code 방식으로 설계한다.
- 사내 데이터 활용 시 보안, 개인정보, 거버넌스 이슈를 고려한다.

## 교육 철학: 위임할 것과 통제할 것

### SQL 다루기 — 문법 암기보다 “읽고 의도 설명하기”

목표는 SQL 개발자 양성이 아니다. 목표는 **AI가 생성한 SQL이 질문에 맞는지 판단할 수 있을 정도의 독해력**을 갖추는 것이다.

| 영역 | AI에게 위임 | 사람이 통제해야 할 것 |
|---|---|---|
| 작성 | 자연어를 SQL 초안으로 변환 | 생성된 SQL이 질문에 맞는지 확인 |
| 정리 | 결과를 표와 요약으로 정리 | 요약이 실제 결과를 왜곡하지 않는지 확인 |
| 반복 | 유사 추출 요청 반복 처리 | 반복 기준이 바뀌지 않았는지 확인 |
| 구조 이해 | — | 테이블, 컬럼, PK/FK, Join 개념 이해 |
| 독해 | — | SELECT/FROM/WHERE/JOIN/GROUP BY를 읽고 의도를 설명 |
| 검증 | — | 없는 컬럼·테이블 환각, 잘못된 날짜·필터·집계 기준 탐지 |
| 맥락 판단 | — | 결과가 실제 비즈니스 맥락과 맞는지 판단 |

핵심 전환은 다음과 같다.

> “수강생을 SQL 작성자로 만들지 않는다”와 “수강생이 SQL을 읽지 못하게 둔다”는 다르다. 후자는 검증을 불가능하게 만든다. 본 과정은 **쓰기는 빼되 읽기는 가르친다.**

---

### 통계 다루기 — 계산보다 “기준을 묻는 사고”

AI는 기초 통계량부터 회귀, 시계열 분석, 시각화까지 많은 작업을 수행할 수 있다. 그러나 계산을 위임한다고 판단까지 위임되는 것은 아니다.

본 과정은 수식 유도와 손계산을 중심으로 통계를 가르치지 않는다. 대신 **기준을 묻는 능력**을 가르친다.

수강생이 갖춰야 할 질문은 다음과 같다.

- 이 평균은 극단치에 왜곡되지 않았는가?
- 평균과 중앙값을 함께 보았는가?
- 두 비교가 같은 기준과 같은 기간으로 이루어졌는가?
- 전월 대비 증가는 실제 성장인가, 계절 요인인가?
- 상관관계를 인과관계로 넘겨짚지 않았는가?
- 표본 수가 너무 적어 해석을 보류해야 하는 상황은 아닌가?
- 이 그래프는 의사결정에 필요한 정보를 왜곡 없이 보여주는가?

“표본 수가 충분한가?”와 같은 체크리스트를 넣으려면, 수강생이 판단할 수 있는 어림 기준도 함께 제공해야 한다. 예를 들어 집단별 관측치가 매우 적을 때는 비율·평균 차이를 강하게 해석하지 않고, 추가 데이터 확보나 해석 보류를 검토한다. “충분한 표본 수”는 분석 목적과 지표에 따라 달라지지만, 입문 과정에서는 **표본이 매우 적을 때 비교 결과를 단정하지 않는다**는 원칙을 먼저 배운다.

## Session 1 — 생성형 AI 데이터 분석의 원리와 한계

### 목표

생성형 AI가 정형 데이터를 어떻게 다루는지 이해하고, 검색·참조와 실제 계산을 구분한다.

### 주요 내용

- 생성형 AI 데이터 분석의 가능성과 한계
- 파일 기반 EDA의 기본 개념
- Text-to-SQL의 위치와 한계
- hallucination이 데이터 분석에서 위험한 이유
- 데이터 보안과 개인정보 보호의 기본 원칙
- CSV, 엑셀, SQL DB의 차이
- 테이블, 컬럼, 행, 스키마, 메타데이터의 개념

### 핵심 개념: RAG/검색 vs 코드 실행/계산

이 과정에서 가장 중요한 구분 중 하나는 **검색·참조**와 **실제 계산**의 차이다.

지식 파일에 CSV를 올리고 instructions만 주는 방식은 대체로 **검색·참조 중심**이다. 이 방식은 문서에서 관련 내용을 찾아 요약하는 데는 유용하지만, 수천 행을 정확히 합산·집계하는 용도에는 부적합하다.

반면 코드 인터프리터, 데이터 분석 기능, 스프레드시트 계산, SQL 실행기처럼 실제 계산이 일어나는 환경은 행 단위 계산과 집계를 수행할 수 있다.

따라서 원칙은 다음과 같다.

> **정확한 수치 집계가 필요한 작업은 검색·참조가 아니라 코드 실행, 스프레드시트 계산, SQL 실행처럼 실제 계산이 일어나는 환경에서 수행해야 한다.**

중요한 것은 “추측하지 말라”는 instruction만으로 수치 계산 오류가 사라지지 않는다는 점이다. 이 문장은 환각을 줄이는 데 도움을 줄 수 있지만, 계산 기능이 없는 환경에서 정확한 집계를 보장하지는 않는다.

### 실습

- 동일한 CSV를 코드 실행이 꺼진 환경과 켜진 환경에 각각 올려 총합·평균을 물어본다.
- 두 결과를 비교하고 왜 차이가 날 수 있는지 토론한다.
- 데이터 구조 설명, 결측치, 이상치 탐색을 요청한다.
- “이 데이터로 어떤 질문을 할 수 있는가?”를 AI와 함께 도출한다.

### 보안 원칙

실제 사내 데이터와 개인정보를 외부 AI 도구에 업로드하지 않는다. 실습은 반드시 가명처리된 mock data로 진행한다. 사내 교육이라면 보안 정책, 데이터 보존 정책, 접근 권한, 개인정보 처리 기준을 사전에 확인한다.

## Session 2 — 데이터 구조와 검증을 위한 최소 독해

### 목표

AI가 생성한 SQL, 표, 통계 결과를 검증하기 위한 최소한의 데이터 구조 독해력을 갖춘다.

### 주요 내용

- 스키마 읽기
- 테이블, 행, 컬럼, 데이터 타입
- primary key와 foreign key
- Join이 필요한 이유
- SQL 독해 입문
- 평균과 중앙값
- 극단치의 영향
- 상관관계와 인과관계의 차이
- 표본이 적을 때의 위험
- 비교 기준의 공정성

### SQL 독해의 최소 목표

수강생은 SQL을 직접 작성하지 않아도, 다음 정도는 말로 설명할 수 있어야 한다.

- 무엇을 가져오는가? `SELECT`
- 어디서 가져오는가? `FROM`
- 어떤 테이블을 연결하는가? `JOIN`
- 어떤 조건으로 거르는가? `WHERE`
- 어떤 단위로 묶는가? `GROUP BY`
- 어떤 순서로 정렬하는가? `ORDER BY`

예를 들어 AI가 생성한 쿼리를 보고 다음 문장으로 설명할 수 있어야 한다.

> “이 쿼리는 2026년 2분기 결제 완료 주문만 대상으로, 제품군별 매출 합계와 주문 수를 계산하는 쿼리다.”

### 실습

- 짧은 SQL 5개를 읽고, 각각 “무엇을 세는 쿼리인가?”를 한국어로 설명한다.
- 의도적으로 틀린 Join, 잘못된 날짜 필터, 존재하지 않는 컬럼을 심어 둔 쿼리를 검토한다.
- 작은 표본으로 계산한 비율 비교가 왜 위험한지 토론한다.
- AI에게 “이 쿼리의 의도를 한 문장으로 설명해줘”라고 요청한 뒤, 설명이 실제 쿼리와 맞는지 대조한다.

## Session 3 — 질문 설계와 Text-to-SQL Prompt Engineering

### 목표

모호한 업무 질문을 분석 가능한 데이터 질문으로 바꾸고, AI가 생성한 SQL을 검증한다.

### 주요 내용

좋은 데이터 질문은 다음 7요소를 포함한다.

1. 대상
2. 기간
3. 지표
4. 필터 조건
5. 비교 기준
6. 출력 형식
7. 의사결정 목적

예를 들어 “최근 매출 어때?”라는 질문은 분석 질문으로 부족하다. 다음과 같이 바꿔야 한다.

> “2026년 4월부터 6월까지 A 제품군의 월별 매출액, 주문 수, 평균 주문 금액을 구하고, 전년 동기 대비 증감률을 표와 선그래프로 보여줘. 매출 기준은 결제 완료 주문만 포함하고, 환불 주문은 제외해줘.”

### 실습

다음과 같은 질문을 분석 질문으로 바꾼다.

- “최근 매출 어때?”
- “우리 고객이 줄었어?”
- “광고비가 효과 있었어?”
- “이번 분기 영업이 잘되고 있어?”
- “어느 부서 비용이 많이 늘었어?”

변환한 질문을 AI에 넣어 SQL 초안을 받고, Session 2의 독해법으로 직접 검증한다.

## Session 4 — 파일 기반 대화형 EDA 실습

### 목표

CSV/엑셀 파일을 기반으로 AI와 대화하며 탐색적 데이터 분석을 수행하고, 생성된 코드와 결과를 검증한다.

### 주요 내용

- EDA의 기본 흐름
- 데이터 구조 파악
- 결측치 탐색
- 이상치 탐색
- 중복값 확인
- 집단별 요약
- 분석 기준을 바꿔 후속 질문하기
- AI가 돌린 코드 또는 SQL을 펼쳐 보고 계산 과정을 확인하기

이 세션의 핵심은 AI를 결과만 받는 oracle처럼 쓰지 않는 것이다.

> “코드를 안 짠다”와 “생성된 코드를 안 본다”는 다르다. 후자는 검증력을 떨어뜨린다.

### 선택 데모

강사가 sqlite 같은 가벼운 샌드박스에서 생성 SQL을 실제로 한 번 실행한다. 이를 통해 “종이 위 SQL”이 아니라, SQL이 실제 결과와 어떻게 연결되는지 1회 시연한다. 수강생이 직접 환경을 구축하지는 않는다.

### 실습

- mock CSV 구조 요약
- 결측치, 이상치, 중복값 탐색
- 기초 통계량 계산
- 그룹별 매출, 고객 수, 전환율 비교
- AI가 제안한 분석 방향 검토 및 수정
- 분석에 사용된 코드나 쿼리의 의도 설명 요청

## Session 5 — 시각화와 비즈니스 해석

### 목표

분석 목적에 맞는 시각화를 선택하고, 결과를 실무 언어로 해석한다.

### 주요 내용

| 목적 | 적합한 차트 |
|---|---|
| 시간에 따른 추세 | 선그래프 |
| 항목 간 비교 | 막대그래프 |

| 구성비 | 누적 막대그래프, 원그래프 |

잘못된 차트 선택은 오해를 만든다. 특히 축 범위 과장, 단위 누락, 범례 오류, 너무 작은 표본의 과도한 시각화는 주의해야 한다.

### 실습

- 같은 데이터를 여러 그래프로 표현하고 비교한다.
- AI에게 “이 차트를 고른 이유”를 설명하게 한다.
- 분석 결과를 다음 세 가지 형식으로 변환한다.
  - 3줄 요약
  - 임원 보고용 요약
  - 실무 액션 아이템
- “그래서 무엇을 해야 하는가?”를 중심으로 후속 질문을 만든다.

## Session 6 — 데이터 분석 Agent 개념과 설계

### 목표

업무 목적에 맞는 데이터 분석 Agent의 구조와 한계를 이해하고, prototype 기획서를 작성한다.

### 주요 내용

Data Agent는 일반적으로 다음 요소로 구성된다.

- LLM
- Memory
- Knowledge
- Tools
- Workflow

일반 챗봇과 맞춤형 Agent의 차이는 “어떤 데이터를 참조하고, 어떤 도구를 사용하며, 어떤 절차를 따르도록 설계되었는가”에 있다.

### Agent 정확도의 핵심

Knowledge만으로는 정확한 집계를 보장하기 어렵다. 수치 집계가 필요한 Agent는 Knowledge만으로는 부족하며, 코드 실행, 스프레드시트 계산, SQL 실행, BI 도구 연동 등 실제 계산 도구를 Tools로 가져야 한다.

즉, Session 1에서 배운 검색·참조와 계산의 구분이 Agent 설계에서도 그대로 적용된다.

### Agent가 해서는 안 되는 일

- 검증 없이 의사결정을 자동화하기
- 민감정보를 외부 도구에 무단 업로드하기
- 데이터에 없는 내용을 추측해서 답하기
- 계산 기능 없이 정확한 수치 집계를 약속하기
- 권한이 없는 데이터까지 조회하도록 설계하기

### 실습: Agent 기획서 작성

각자 자신의 업무를 기준으로 간단한 Agent 기획서를 작성한다.

기획서 항목은 다음과 같다.

- 해결하려는 업무 문제
- 사용할 데이터 종류
- 사용자가 물어볼 질문 예시
- Agent가 제공해야 할 답변 형식
- 필요한 데이터 보안 수준
- 사람이 반드시 검토해야 할 지점
- 수치 계산이 필요한가?
- 계산 도구가 필요한가?
- 실제 사내 적용 시 추가로 필요한 것은 무엇인가?

## Session 8 — No-Code Workflow 템플릿 실습과 최종 발표

### 목표

완성된 no-code workflow 템플릿을 일부 수정해보고, 각자의 Agent prototype을 발표한다.

### 주요 내용

Dify, Coze 등 no-code agent builder는 workflow 사고방식을 익히는 데 유용하다. 다만 반나절 안에 비전공자가 노드를 처음부터 쌓는 것은 얕은 데모로 끝나기 쉽다. 따라서 입문 과정에서는 강사가 제공한 완성 템플릿을 일부 수정하는 방식이 적절하다.

기본 workflow는 다음과 같다.

1. 사용자가 자연어로 질문한다.
2. Agent가 질문의 의도를 파악한다.
3. Agent가 참조 가능한 스키마를 확인한다.
4. Agent가 SQL 초안을 생성한다.
5. Agent가 SQL의 의도와 조건을 설명한다.
6. 사용자가 검토하거나 승인한다.
7. 쿼리 결과를 받아온다.
8. Agent가 결과를 자연어로 해석한다.
9. 필요하면 후속 질문을 제안한다.

### 실습

- 강사가 제공한 workflow 템플릿 확인
- 스키마 참조 노드 수정
- SQL 생성 prompt 수정
- 결과 해석 prompt 수정
- fallback 답변 추가
- “데이터에 없는 내용은 답하지 않기” 규칙 추가
- 수치 집계가 필요한 질문에 계산 도구가 연결되어 있는지 확인

### 주의

입문 과정에서는 실제 사내 DB 연결이 아니라 **mock schema와 샘플 CSV 기반 prototype**을 목표로 한다. 실DB 연동, 권한 관리, 로그 관리, API key 관리, 배포는 심화 과정에서 다루는 것이 적절하다.

### 최종 발표 항목

- 어떤 업무 문제를 푸는 Agent인가?
- 어떤 데이터를 사용했는가?
- 사용자는 어떤 질문을 할 수 있는가?
- Agent는 어떤 방식으로 답하는가?
- 답변의 한계는 무엇인가?
- 특히 수치 정확도와 관련된 한계는 무엇인가?
- 실제 사내 적용을 위해 추가로 필요한 것은 무엇인가?
- 보안과 거버넌스 측면에서 주의할 점은 무엇인가?

## 부록 A — Prompt Template

### 1. 데이터 구조 파악

```text
이 파일의 데이터 구조를 설명해줘.
각 컬럼의 의미를 추정하고,
데이터 타입, 결측치, 이상치 가능성, 분석에 주의해야 할 점을 표로 정리해줘.
확실하지 않은 내용은 추측이라고 표시해줘.
```

### 2. 좋은 분석 질문으로 바꾸기

```text
다음 업무 질문을 데이터 분석에 적합한 질문으로 바꿔줘.
분석 대상, 기간, 지표, 필터 조건, 비교 기준, 출력 형식,
의사결정 목적을 명확히 해줘.
```

### 3. Text-to-SQL

```text
아래 스키마를 기준으로 사용자의 질문에 답하기 위한 SQL을 작성해줘.
존재하지 않는 테이블이나 컬럼은 만들지 말고,
필요한 경우 먼저 확인 질문을 해줘.

SQL을 제시한 뒤,
어떤 테이블을 어떤 조건으로 사용했는지 설명해줘.
```

### 4. 수치 계산

```text
이 집계는 추정하지 말고 코드 실행으로 정확히 계산해줘.
계산에 사용한 행 수, 필터 조건, 집계 기준을 함께 보고해줘.
```

### 5. 결과 검증

```text
방금 생성한 SQL 또는 분석 결과가 사용자의 원래 질문에 정확히 답하는지 검토해줘.
날짜 기준, 필터 조건, 집계 기준, Join 조건, 누락 가능성,
과도한 추정 여부를 점검해줘.
```

### 6. 비즈니스 해석

```text
이 분석 결과를 실무자가 이해할 수 있는 비즈니스 언어로 해석해줘.
핵심 발견 3개, 가능한 원인 3개, 추가로 확인해야 할 데이터 3개,
실행 가능한 액션 아이템 3개로 정리해줘.
```

### 7. 보고서 초안

```text
위 분석 결과를 바탕으로 팀장 보고용 1페이지 요약 보고서 초안을 작성해줘.
구성은 배경, 분석 방법, 핵심 결과, 해석, 권장 액션,
한계와 추가 확인 사항으로 해줘.
```

질문 예시:

- 이번 분기 예상 매출은 얼마인가?
- 영업 단계별 병목은 어디인가?
- 담당자별 계약 전환율은 어떻게 다른가?
- 계약 가능성이 높은 고객군은 어디인가?

### 재무

데이터 예시:

- 매출
- 비용
- 부서별 예산
- 실제 지출
- 계정 과목
- 월별 손익
- 현금 흐름

질문 예시:

- 최근 6개월간 비용이 가장 많이 증가한 항목은 무엇인가?
- 예산 대비 초과 지출이 발생한 부서는 어디인가?
- 월별 영업이익률 추이는 어떠한가?
- 특정 비용 항목이 전체 비용에서 차지하는 비중은 얼마인가?

### 인사

데이터 예시:

- 부서
- 직급
- 입사일
- 퇴사일
- 근속연수
- 교육 이수 여부
- 평가 등급
- 익명화된 만족도 점수

질문 예시:

- 부서별 평균 근속연수는 어떻게 다른가?
- 최근 1년간 퇴사율이 높은 부서는 어디인가?
- 교육 이수 여부와 평가 결과 사이에 관계가 있는가?
- 직급별 인원 구성은 어떻게 되어 있는가?

인사 데이터는 개인정보와 민감정보가 포함될 가능성이 높으므로 반드시 익명화 또는 가명처리된 데이터를 사용해야 한다.

## 부록 D — 강의 직전 재확인 사항

ChatGPT, Custom GPTs, Claude, Dify, Coze 등 AI 도구의 기능은 빠르게 바뀐다. 특히 다음 항목은 강의 직전 반드시 다시 확인한다.

- 파일 업로드 가능 여부
- CSV/엑셀 처리 방식
- 코드 실행 또는 데이터 분석 기능 지원 여부
- 요금제별 제한
- 데이터 보존 정책
- 기업용 계정의 보안 설정
- 외부 도구 연동 가능 여부
- no-code workflow 기능 변화
- API key와 로그 관리 정책

도구 관련 설명은 작성 시점의 일반적 개념을 바탕으로 한 것이므로, 실제 강의 설계는 각 도구의 최신 사양을 확인한 뒤 확정해야 한다.

## 과정 설계의 핵심 성공 조건

이 과정의 성공 여부는 도구 자체보다 **실습 데이터, 업무 맥락, 검증 커리큘럼**에 달려 있다.

교육 전 다음 사항을 준비하는 것이 좋다.

- 수강생의 주요 직무 파악
- 직무별 mock data 준비
- 실제 업무 질문에 가까운 실습 과제 설계
- 보안상 안전한 데이터 사용
- AI 분석 결과를 검증하는 체크리스트 제공
- 생성된 코드와 SQL을 읽는 최소 독해 훈련
- 최종 프로젝트를 통한 업무 적용 가능성 확인

결국 이 과정의 핵심은 “AI가 데이터를 분석해준다”가 아니다. 핵심은 수강생이 **더 좋은 질문을 만들고, 데이터 구조를 이해하고, 생성된 코드와 수치를 읽어 검증하고, 실제 의사결정으로 연결**하게 되는 것이다.
